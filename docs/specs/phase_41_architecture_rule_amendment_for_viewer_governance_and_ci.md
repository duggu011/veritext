## Goal

Decide, explicitly and auditably, whether Veritext should amend its current
architecture rules before any viewer, governance, or CI work begins.

Phase 41 is a policy and workflow phase, not a product implementation phase. It
must either:

- amend `AGENTS.md` and `CLAUDE.md` with byte-identical, narrowly scoped
  allowances for later approved phases, or
- explicitly defer those roadmap items and leave the existing bans in place.

The phase exists because the roadmap's HTML provenance viewer, review UI, and
CI integration conflict with current architecture rules unless a prior phase
approves a narrow exception.

## Non-Goals

- Do not build the HTML provenance viewer.
- Do not build a review UI, web UI, local web server, REST API, or browser app.
- Do not add CI/CD configuration, GitHub Actions workflows, Docker, deployment
  packaging, vector databases, embeddings, local model serving, or agent
  frameworks.
- Do not change extraction behavior, prompts, LLM routing, report schemas, audit
  schemas, config defaults, or evaluation thresholds.
- Do not approve any architecture-rule amendment by implication. The amendment
  text must be explicit, byte-identical in `AGENTS.md` and `CLAUDE.md`, and
  accepted by the operator before implementation.
- Do not use Phase 41 to weaken I1-I9, provenance requirements, source-offset
  invariants, Pydantic stage contracts, or auditability rules.

## Domain-Scope Alignment

Phase 41 stays domain-neutral. Any future viewer or governance surface must help
auditors inspect exact source spans, prompt hashes, config hashes, schema
hashes, signed manifests, rejection trails, and run diffs across all target
domains.

The architecture decision must not privilege legal, financial, clinical,
insurance, standards, scientific, procurement, or audit-evidence documents. It
only defines whether later phases may add specific non-extraction surfaces.

## Current State

The active architecture rules currently ban:

- web UI
- REST API
- Docker
- CI/CD
- vector DBs
- embeddings
- fine-tuning hooks
- local model serving
- agent frameworks

Phase 40 added non-UI signed manifests, audit integrity events, confidence
buckets, and deterministic run diffs. Those artifacts are enough substrate for a
future static provenance viewer, but current rules still ban web UI and CI/CD.

`docs/phase_26_plus_roadmap.md` defines Phase 41 as the required decision point
before Phase 42 HTML provenance work. It also lists architecture conflicts for
HTML viewer/review UI, CI integration, local model support, non-SQLite audit
backends, Docker, REST APIs, fine-tuning hooks, vector DBs, embeddings, and
open-web crawling.

## Decision Options

The board for Phase 41 should resolve each option explicitly.

### Static Generated HTML

Option A: allow only deterministic static HTML files generated by the reporter,
with no server, no REST API, no client-side data fetching, no persistent local
state, and no extraction behavior changes.

Option B: keep all HTML viewer work deferred and preserve the current web UI
ban unchanged.

Option C: allow a broader review UI. This option conflicts with the current
architecture rules more deeply and should require a separate later product
architecture phase if selected.

### CI Integration

Option A: allow repository CI only for existing deterministic checks such as
unit tests, lint, smoke, prompt-neutrality checks, and explicit evaluation
suites, with no deployment packaging and no secret-dependent external services.

Option B: keep all CI/CD deferred and preserve manual gate execution.

Option C: allow broader CI/CD or deployment workflows. This conflicts with the
current Docker/deployment packaging bans and should require a separate later
operations architecture phase if selected.

### Governance Workflow

Option A: allow file-based or audit-DB-backed governance records without a UI,
such as review decision contracts, reviewer IDs, decision timestamps, and
review-status reports.

Option B: defer all governance workflow implementation.

Option C: allow a human review UI. This overlaps with web UI and should not be
approved unless the static-vs-UI boundary is rewritten explicitly.

## Proposed Conservative Default

If the operator wants a minimal amendment, the conservative default is:

- allow deterministic static HTML report artifacts only as files written by the
  reporter in a later approved phase;
- keep web servers, REST APIs, dynamic browser apps, external services, Docker,
  vector DBs, embeddings, and agent frameworks banned;
- allow CI only for deterministic repository verification if the operator
  explicitly approves CI in Phase 41;
- defer review UI and active-learning/fine-tuning behavior.

This default preserves the extraction kernel's current architecture while
unblocking the smallest practical provenance-review surface.

## Approved Resolutions

The operator approved the conservative default on 2026-05-30.

- Deterministic static HTML report artifacts are allowed only in later approved
  reporter phases, as local files with no web server, REST API, client-side data
  fetching, persistent browser/local state, dynamic browser app, or extraction
  behavior changes.
- Repository CI is allowed only in later approved phases for deterministic
  repository verification such as unit tests, lint, smoke, prompt-neutrality
  checks, and explicit evaluation suites. Deployment packaging and
  secret-dependent external services remain banned.
- Non-UI governance records are allowed only in later approved phases as
  file-based or SQLite/audit-DB-backed records. Human review UI,
  active-learning behavior, and fine-tuning behavior remain deferred.
- The exact architecture-rule amendment must be added byte-identically to
  `AGENTS.md` and `CLAUDE.md`.
- Phase 42 should be renamed from "HTML Provenance Viewer, If Approved" to
  "Static Provenance Artifact, If Approved" to avoid web UI ambiguity.

## Invariant Impact

Phase 41 must not weaken I1-I9. If any amendment text could reduce exact
source/offset provenance, report determinism, Pydantic stage contracts, audit
logging, or no-silent-drop behavior, stop and redesign the amendment.

Any later viewer or governance phase must consume existing contracts and audit
records rather than becoming an alternate extraction path.

## Prompt Impact

No prompt files may change in Phase 41.

## Configuration Impact

No runtime configuration may change in Phase 41.

## Tests and Verification Gates

Required checks for this doc-only phase:

- `cmp -s AGENTS.md CLAUDE.md`
- `git diff --check`
- `git diff --exit-code -- prompts`
- draft-marker and open-question scan on the Phase 41 spec and board
- `rg` checks proving the selected amendment text is present in both agent
  files when an amendment is approved

If Phase 41 only defers all amendments, the verification gate should prove the
existing architecture-rule bans remain intact.

`make test`, `make lint`, `make smoke`, and evaluation gates are optional for
the final Phase 41 gate unless the phase unexpectedly touches executable code.
